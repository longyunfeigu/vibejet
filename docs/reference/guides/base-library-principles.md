# 基础库原则：让 AI 在此之上开发而不跑偏

> 本仓库不只是一个全栈应用，更是一个**基础库 / 模板**：沉淀常用能力和经过验证的参考代码，
> 让 AI 从 0 到 1 生成代码时有可依赖的基线，减少随机漂移。
> 这份文档是该目标的**基线原则 + 策展 checklist**——它本身就是被沉淀下来、AI 每次能读到的约束。

## 0. 命题纠正

常见假设："有参考代码 → AI 不跑偏"。准确的说法是：

> **参考代码是必要的，但远不够。它压住的是「结构性漂移」（目录、分层放错），
> 压不住「语义性漂移」（逻辑错、复制而非复用、过度设计）。而且它在消除一种随机性的同时，
> 引入了新的随机性：AI 该选哪个参考。**

基础库真正的敌人**不是「缺代码」，是「信号有歧义 + 信号腐烂」**：

- 一段好代码，不会让 AI 不跑偏。
- **两段都像对的、或一段过时却看着权威的**代码，一定让它跑偏。

## 1. 约束力排序（地基）

治漂移靠的东西，从强到弱：

1. **会失败的机器检查**（lint / 架构测试 / schema 校验 / pre-commit gate）——AI 骗不过去。
2. **一键验证**（能跑起来 + 冒烟 e2e）——逼 AI 用真实行为证明「做完了」。
3. **散文规则**（`AGENTS.md` 里的文字）——最弱，长会话里最先被忘。

推论：任何停留在第 3 层的不变量，在 AI 高频改动下都会腐烂。有价值的规则必须往第 1 层搬。

**但有一个前置步骤：给规则上锁之前，先用第一性原理问「这条规则还该存在吗」。**
案例：文件头注释规则（曾在 `.claude/rules/doc-maintenance.md`）约半数文件不遵守，
第一反应是加 pre-commit gate 强制——加完才复盘发现规则本身不成立：`input/output/pos`
全是**可从代码推导的事实**（import 即 input、公开符号即 output、DDD 路径即架构位置），
维护 165 份分布式缓存必然腐烂，而腐烂的注释比没有更糟。最终规则与 gate 一并删除。
教训：**「规则没被遵守」有两种解释——缺强制，或规则本身成本收益不成立。先排除后者，
否则你是在给腐烂的东西上锁。**

## 2. 让基础库真正生效的做法

1. **每种能力只留一个 canonical，并机器可读地声明它。**
   用 frontmatter 的 `status: canonical | experimental | deprecated`，而不是靠文字描述。
   病根不是「有 N 个重叠实现」，是「没有字段告诉 AI 哪个是当前的」。
2. **把散文规则翻译成会失败的代码。**
   DDD 依赖方向用 import-linter 拦；头注释用 pre-commit 拦；「公共 API 变了要更新契约」用 diff 检查。
   已落地（2026-07 第一批）：DDD 依赖方向 import-linter 四条契约 + `print` 禁令
   flake8-print + UoW 形状 `backend/tests/test_architecture.py`；
   设计与被拒绝的方案见 `docs/superpowers/specs/2026-07-02-architecture-rule-locking-design.md`。
3. **断言「当前状态」的文档必须生成或被校验，不能手维护。**
   断言型文档（README 索引、「当前架构」、状态清单）一旦和现实矛盾，AI 会信文档——这是最毒的腐烂。
4. **弃用 = 删除或硬隔离，不留墓碑。**
   「拒绝运行 + 打印已弃用」对人有用，但 AI 读不出「死」的语义，照抄不误。要删，
   或进 `docs/archive/`（被 `AGENTS.md` 明确声明为非基线的隔离区）。
5. **沉淀「判断」而不只是「代码」。**
   commit trailers 的 `Rejected:` / `Constraint:` 告诉 AI「什么时候不要这样写」，比代码本身更能防过度设计。
6. **用脚手架生成器收窄 AI 的决策面。**
   能被生成器固定的结构（如「新增实体 N 步」），就别让 AI 每次重新决策，让它只填业务逻辑。
7. **留一条能一键跑起来的垂直切片 + seed 数据**，给 AI 真实锚点，胜过十段孤立 snippet。
8. **不沉淀「可从代码推导的事实」。**
   文档只承载不可推导的意图（为什么、拒绝了什么、约束与不变量）；凡是 AI 读代码 10 秒能推出的
   （依赖、导出、所在层），写成注释/文档就是一份没有失效机制的缓存，必然腐烂。
   owner 归属这类平台性需求用平台标准解（CODEOWNERS），不要散在文件头里自建。

## 3. AI 在此过程最容易犯的错

1. **把过时/弃用模式当 canonical 抄**——因为没有信号告诉它哪个是当前的。
2. **传播基础库自己的缺陷**——库里用了 `print()`、缺 ownership 校验的 pattern，会被 AI 忠实放大。
3. **复制而非复用**——把好代码拷到新地方再改，制造分叉，而非 import / extend。
4. **仪式化套用分层**——为两方法功能凿出全套空壳、贫血转发 service，把手段当 KPI。
5. **只增不删的堆积**——单个都「挺好」，合起来落入倒 U 型右半段。
6. **自信地遵循已腐烂的文档/注释**——比没有更糟，因为看着权威。

## 4. 决定成败的那一条

**基础库的收益是倒 U 型，不是线性的。** 稀缺资源是 **AI 的注意力 + 无歧义的 canonical 信号**，
不是例子的数量。塞太多「好代码」→ context 膨胀、模式冲突、AI 分不清 canonical 与历史包袱。

> **治 AI 漂移的解药不是往库里加更多好代码，而是「更强的约束 + 每种能力唯一 canonical +
> 自动化的腐烂检测」。**「沉淀」这个动作自带失败模式——只沉淀不策展，产出的就是一堆需要清理的残留。
> 杠杆在「减法 + 上锁」，不在「加法」。

## 5. 策展 Checklist（新增/清理内容前逐条过）

**加内容前：**
- [ ] 这个能力**已经有** canonical 实现/skill 了吗？若有 → 扩展它，不要新建第二个。
- [ ] 新增的 skill / 模式 frontmatter 有没有 `status`？是 `canonical` 还是 `experimental`？
- [ ] 它引入的不变量，有没有一个**会失败的检查**兜底（而不是只写进散文）？
- [ ] 示例代码有没有违反本仓库红线（`print`、缺 ownership、复制粘贴）？会被 AI 抄的地方尤其严查。
- [ ] 要沉淀的信息是「可从代码推导的事实」还是「不可推导的意图」？前者不要写成文档/注释。

**给规则加强制前：**
- [ ] 这条规则从第一性原理**还成立吗**？「没被遵守」可能意味着该删而不是该锁。

**清理时：**
- [ ] 弃用的东西是**删了 / 进 archive** 了，还是留了「拒绝运行」的墓碑？墓碑要清。
- [ ] 有没有**引用已删资产**的死链接 / 死脚本 / 死文档？
- [ ] 断言「当前状态」的文档（README 索引等）和现实一致吗？
- [ ] 同一能力是否存在 v1/v2、或多个 description 相近的重叠实现？只留一个 canonical。

**判断存活性时：**
- [ ] 记住：**靠语义匹配触发的资产（skill/prompt），存活性不能靠引用计数判断**——
      没人引用也会被调用。用显式 `status` 元数据，不要用「零引用 = 可删」。
