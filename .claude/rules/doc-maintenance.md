# 文档维护规则

当你修改或创建代码文件时，**必须**执行以下步骤：

## 0. 先读取现有文档（重要！）

修改文件前，**必须先检查并读取**：
- 所属文件夹是否已有 README.md？**如果有，先读取它**
- 文件是否已有头部注释？**如果有，保留并更新**

⚠️ **禁止直接覆盖创建**！

## 1. 文件头部注释

每个代码文件必须在文件开头包含标准四行注释（紧跟在 shebang/编码声明之后；如存在模块 docstring，注释置于 docstring 之前）：

Python / Shell：

```text
# input: {输入描述}
# output: {输出描述}
# owner: {所有者}
# pos: {架构位置描述}；一旦我被更新，务必更新我的开头注释以及所属文件夹的md
```

其他语言 / proto：

```text
// input: {输入描述}
// output: {输出描述}
// owner: {所有者}
// pos: {架构位置描述}；一旦我被更新，务必更新我的开头注释以及所属文件夹的md
```

要求：
- `input`：本文件依赖的外部输入/系统（如：DB、Redis、第三方 API、环境变量、配置文件等）
- `output`：本文件对外提供的能力（如：公开函数/类、HTTP/gRPC 接口、任务、模型等）
- `owner`：文件所有者（GitLab 用户名，用于 PR @mention）
- `pos`：本文件在本系统局部的职责/位置（例如：API 路由、应用服务、领域实体、仓储实现）
- `pos` 行末必须包含：`一旦我被更新，务必更新我的开头注释以及所属文件夹的md`
- 头注释必须"具体到文件"：同一目录下不同文件的 `output/pos` 不应长期保持完全相同（至少包含该文件的核心符号/职责差异）

### owner 字段规则

**新建文件时**：
1. 从环境变量 `CLAUDE_FILE_OWNER` 读取当前用户的 GitLab 用户名
2. 如果环境变量未设置，使用 `unknown` 并提醒用户在 `.claude/.env` 中配置

**更新文件时**：
- **保留原有 owner**，不要修改
- 如果原文件没有 owner 字段（旧格式三行注释），**不要添加**（保持向后兼容）

**示例**（新建 Python 文件）：
```python
# input: 环境变量 (.env), DATABASE__URL
# output: settings 全局配置对象, Settings 配置类
# owner: wanhua.gu
# pos: 核心配置 - 配置管理；一旦我被更新，务必更新我的开头注释以及所属文件夹的md
```

## 2. 文件夹 README.md

修改文件后，更新所属文件夹的 README.md：
- 如果已存在：先读取，增量更新文件索引
- 如果不存在：创建新文件

## 3. 上级目录传播

检查上级目录的 README.md 是否需要更新。

## 4. 文档维护报告

完成后输出报告。
