# wechat-article

把一个项目或产品变成一篇"有人想转发"的公众号文章。

这个 skill 不是套模板填空。它先理解项目做了什么、解决什么问题，然后按爆款文章的结构和节奏写出来。

## 适用场景

- 用户说"写公众号文章""宣发文档""推广文章"
- 用户有一个开源项目或技术产品，想写介绍文章
- 用户想重写/优化已有的推广文章

## 不适用场景

- 纯技术文档（README、API 文档）
- 新闻稿、PR 稿（需要严肃正式的语气）
- 用户要的是短文案（朋友圈、推文）而非长图文

## 输入

从用户输入或项目代码库中提取：

- **项目是什么**：核心功能、技术栈、差异化
- **解决什么问题**：目标用户的痛点和场景
- **为什么做这个**：动机、背景故事
- **特殊卖点**：AI Coding 开发过程、技术亮点、开源信息
- **已有素材**：截图、demo、竞品对比

如果信息不足，**主动向用户追问**，不要瞎编。

## 写作方法论

### 核心原则

1. **"我在用"视角，不是"它能做"视角**。全程第一人称叙事，像跟朋友安利自己做的东西
2. **功能从使用过程中自然浮现**，不要罗列"功能一、功能二、功能三"
3. **诚实不吹**。主动说不足和局限，读者信任感远比功能清单重要
4. **短段落**。每段 1-3 句话，适配手机阅读。长段落是完读率杀手
5. **加粗关键句**。让"扫读"的人也能抓到核心信息

### 文章结构（五段式）

```
§1 Hook（前 100 字定生死）
   ├─ 蹭热点 / 名人 / 数据 / 个人翻车经历
   ├─ 制造认知冲突或信息缺口
   └─ 一句话引出"所以我做了这个"

§2 先看效果，不讲概念
   ├─ 用一个完整的真实使用场景走一遍
   ├─ 展示具体的输入和输出（截图穿插）
   └─ 让读者先"看到"产品在运行，再解释原理

§3 逐层展开核心功能（2-3 个）
   ├─ 每个功能用"我发现……"引出，不是"本产品支持……"
   ├─ 配截图，图说合一
   └─ 每个功能段落末尾用加粗句总结价值

§4 诚实评价 + 差异化
   ├─ 主动说目前的不足和局限
   ├─ 与同类产品/方案做对比（可用表格）
   └─ 一句话总结核心差异

§5 收尾（开源 + 动机 + 互动）
   ├─ GitHub 链接 + 快速开始命令
   ├─ 为什么做这个（感性但克制）
   ├─ 互动提问引导留言
   └─ 社交钩子（加微信/进群/Star）
```

### 标题公式

好标题 = **数字冲击 + 具体画面 + 好奇缺口**

模板参考：
- `[数字] + [动作] + [意外的结果]`：「1 人 1 天，我用 AI 写了个"开会挨打模拟器"」
- `[痛点场景] + [解决方案暗示]`：「开会总被怼到哑口无言？我花一天做了个 AI 让你提前挨打」
- `[反常识判断]`：「所有人都在用 AI 写代码，但没人用 AI 练过开会」
- `[名人/热点] + [我的延伸]`：「Karpathy 说应该有个产品，我真的去做了」

规则：
- 25-35 字之间
- 口语化，像朋友在群里分享
- 避免"震惊！""万万没想到"等营销号词汇
- 至少生成 5 个候选标题供用户选择

### 情绪曲线设计

文章的情绪节奏应该是：

```
共鸣（痛点）→ 好奇（有解法？）→ 惊喜（居然做到了）→ 信任（诚实说不足）→ 行动（我也试试）
```

不要全程高能吹捧，也不要全程平铺直叙。**在"惊喜"之后放"诚实"段落，是建立信任的关键。**

### 配图策略

如果项目有前端界面，建议截取：

| 优先级 | 截图内容 | 作用 |
|--------|---------|------|
| P0 | 核心功能运行中的画面 | 让读者"看到"产品在工作 |
| P1 | 关键的数据/可视化输出 | 证明产品确实能产出有价值的东西 |
| P2 | 创建/配置界面 | 展示上手门槛低 |
| P3 | 全局界面 | 给读者整体印象 |

图片规则：
- 图文交织，不要文末集中放图
- 每张图上方有引导文字（"接下来发生的事很有意思——"），下方有解读
- 截图前先清理测试数据，确保画面干净

### 语气校准

| 应该 | 不应该 |
|------|--------|
| "说实话这个功能帮助最大" | "本产品的核心竞争力是" |
| "目前有个问题" | "未来规划中将会优化" |
| "我做了一个实验" | "经过严谨的测试验证" |
| "拿这份画像去跑模拟" | "将画像配置导入系统后" |
| "挺有意思" | "令人振奋" |

### 字数控制

- 目标：**2000-3000 字**
- 超过 3000 字完读率明显下降
- 如果内容确实丰富，宁可砍掉次要功能也不要写成万字长文
- 每个功能段落不超过 300 字

## 输出

1. **5 个候选标题**（按风格分类，附推荐理由）
2. **正文 Markdown**（含图片占位或真实图片引用）
3. **摘要**（80 字以内，用于公众号文章摘要/朋友圈分享文案）

输出文件保存到 `docs/reference/research/wechat-launch-article.md`。

## 工作流

```
1. 读取项目信息（README、CLAUDE.md、关键源码）
2. 向用户确认：目标读者、核心卖点、特殊故事（如 AI Coding 经历）
3. 如果需要截图且项目有前端 → 启动服务 + Playwright 自动截图
4. 生成 5 个候选标题，用户选择
5. 按五段式结构写正文
6. 用户审稿 → 迭代修改
```
