# Sharpen Idea(磨点子)

## Overview

把模糊的产品想法磨成尖锐的定位。核心原理:**通用工具很难赢,用户买单的不是功能,而是被解决的恐惧和焦虑。**

流程:三层需求判断 → 横向切分人群 → 纵向深挖场景 → 价值重构。产出一个重构后的产品概念,然后交棒给 `validate-idea` 验证付费意愿。

本 skill 全程用中文与用户对话,**一次只问一个问题**,像教练一样逼问,而不是替用户编造答案。

## When to Use

- 用户说"我想做个XX"但目标人群是"所有人"
- 用户的想法是一个功能(记账、番茄钟、卡路里识别),市面上已有成熟产品
- 用户不确定点子是否值得投入

**不要用于:** 已有清晰细分定位、只差验证付费意愿的点子(直接用 validate-idea);纯技术选型或实现问题。

## 流程

### Step 1:接收原始想法,判断需求层级

让用户用一句话描述想法。然后判断它当前解决的是哪一层需求:

| 层级 | 本质 | 信号 | 付费意愿 |
|------|------|------|----------|
| 痛点 | 恐惧驱动,不解决会很难受 | 用户已在花钱/花时间笨拙地解决 | 高 |
| 爽点 | 即时满足 | 用得爽但免费替代多 | 中,留存好但难收费 |
| 痒点 | 想变更好,可有可无 | "有也挺好",不用也没损失 | 低 |

**警惕:把痒点包装成痛点**("记录喝水让你更健康"——不记录也不会不健康)。如果是痒点,直说,并引导用户往下走——细分之后痒点常常能变成某个人群的痛点。

### Step 2:横向切分人群

不服务"所有人",找到"最痛的那个人群"。

1. 引导用户列出 3-5 个可能需要这个东西的**具体**细分人群(不是"上班族",是"留学生家长"、"产后妈妈"、"小餐馆老板")
2. 逐一评估,填这张表(评估维度详见 reference.md §3):

| 人群 | 痛点强度 | 付费意愿 | 市场规模 | 竞争程度 | 你对人群的理解 |
|------|----------|----------|----------|----------|----------------|

**填表分工:**"痛点强度、付费意愿、你对人群的理解"必须问用户;"市场规模、竞争程度"可由 Claude 基于自身知识评估并注明来源。每个人群必问"他们现在怎么解决这个问题?",答案对照 reference.md §2 替代方案机会判断表来评估竞争程度。本步骤可**按人群打包提问**(一个人群的几个维度一起问),不算违反"一次一问"。

3. 推荐选择:痛点最痛 × 付费意愿最强 × 用户最理解 × 竞争相对不激烈的人群。**用户自己(或身边人)就是该人群,是重要加分项。**

### Step 3:纵向深挖场景

对选中人群,**逐场景提问用户**(他们一天中哪些时刻会被这个问题困扰?具体发生了什么?),由 Claude 把回答整理成"一天时间线"(格式参考 reference.md §4),每个场景标注:

- **痛点**:具体卡在哪
- **情绪**:恐惧 / 焦虑 / 无助 / 孤独 / 愤怒 / 后悔

关键认知:**痛点 = 场景中的情绪 + 付费意愿**,不是功能缺失。产后妈妈的痛点不是"没有健身课程",而是"产后恢复的恐惧和焦虑"。好的产品解决情绪,情绪背后才是付费动力。

如果用户答不上来场景细节,这本身就是红灯——说明他不够理解这个人群,建议先去访谈 3 个真实用户再回来。

### Step 4:价值重构

基于场景和情绪,把"一个功能"升级为"一个解决方案":

- 从"工具"升级为"助手 / 管家 / 教练 / 伙伴"(案例库见 reference.md §5)
- 输出重构后的产品概念:**核心定位一句话 + 3-5 个核心功能 + 初步商业模式 + 竞争壁垒**(护城河类型见 reference.md §6)
- 给出对比表:

| 维度 | 原始想法 | 重构后 |
|------|----------|--------|
| 目标用户 | | |
| 解决痛点 | | |
| 竞争壁垒 | | |
| 付费意愿 | | |

### Step 5:交棒验证

重构完成 ≠ 验证通过。明确告诉用户下一步:

> 概念磨好了,但"用户会不会真掏钱"只能用钱验证。建议接着运行 `/validate-idea` 做付费验证(10 个用户访谈 + 定金测试,定金转化率 >10% 才值得投入),之后用 `/mvp` 规划最小可行产品。

## Quick Reference

- 优先级:痛点 > 爽点 > 痒点
- 越细分,付费意愿越强;通用工具很难赢
- 痛点 = 场景情绪 + 付费意愿,不是功能缺失
- 访谈话术、人群评估维度、替代方案机会判断表 → 见 reference.md

## Common Mistakes

| 错误 | 纠正 |
|------|------|
| 替用户编造人群和场景细节 | 只能引导提问,细节必须来自用户的真实观察;答不上来就让他去访谈 |
| 直接照搬 reference.md 里的示例结论(如记账APP→留学生家长) | 示例仅作格式参考,人群和场景必须由用户真实推导得出 |
| 一次抛出全部问题 | 一次一个问题,等回答再继续 |
| 在 Step 1 就否决痒点想法 | 痒点经过细分常变成某人群的痛点,先走完横切再下结论 |
| 重构后直接建议开始开发 | 必须交棒 validate-idea,付费验证前不写代码 |
| 用"你会用这个产品吗"提问 | 假阳性率极高,用 reference.md 里的行为提问法 |
