# vj-test — Epic 级全自动 E2E 测试

在 `vj-work` 把一个 epic 的实现都落地后，为它做**跨层端到端测试**。

**工作流位置**：在 vj-work 实现完成后执行，沉淀走 vj-compound；完整链条见 `docs/reference/guides/ai-workflow.md` §1。

**设计取舍**：借 ln 的测试规划/执行思想（ln-520 编排、ln-523 风险驱动规划、ln-404 风险约束执行）的**理念**，按 vj 轻量风格自包含实现，**零插件耦合**；前端 E2E 用项目已有的 `web-access` skill 驱动真实浏览器，不引入 Playwright/Puppeteer 依赖。

## 铁律

- **全自动，人不做手工测试**，只读报告。**不设任何人工阻塞闸。**
- **绿 ≠ 有效**：人不验证 → AI 测试是唯一安全网。每个高优先级**代码级**测试必须过**变异自检**（故意改坏目标行为 → 确认测试变红 → 还原）；一直绿的测试 = 测不到东西，丢弃或标记。这是替代人工闸的自动信任机制。
- **断言真实状态、真实链路**：DB 行、缓存条目、API 信封、前端渲染都断言真值；跨层**不 mock**。
- **不重复** vj-work 已写的 per-task 单元/集成测试；vj-test 专做**跨 task 的 E2E 流程**。
- **RBT**：只覆盖高优先级场景，砍低价值测试（不测框架/库/DB 内部，不无脑追覆盖率）。

## 输入

```
vj-test epic-1        # 按 epic 编号
vj-test               # 取最近执行的 epic（其 _ledger 处于 in-progress/done）
```

## 配置项

```yaml
epic_test_runner:
  work_dir: docs/tasks/work/epic-{N}-{slug}/                 # 读 task 文档的 AC；写 _test-plan.md / _test-report.md
  evidence_dir: docs/tasks/work/epic-{N}-{slug}/evidence/    # 前端 E2E 成功截图（如不想入库可 gitignore）
  browser_skill: web-access                                  # 前端 E2E 用的真实浏览器 skill
  app_launch: auto                                           # 起 dev server 的方式（auto 探测 pnpm dev / uv run，或填具体命令）
```

## 🛑 安全边界与降级处置

「全自动、无人工闸」≠「无边界自主」。下表是 skill **会自我设限 / 主动交还给人**的硬边界——🛑 标记处**必停或必降级并写进报告，不静默继续**：

| 触发条件 | 一线处理 | 🛑 仍失败兜底 |
|---------|---------|--------------|
| 测试 DB / API 客户端缺失 | 后端层降级为人工待办 | 🛑 报告写「后端 DB 测试未覆盖」 |
| 缓存不可用 | 缓存断言降级为跳过 | 🛑 报告标注「缓存层未覆盖」 |
| web-access 经 Skill tool 报 not found | 前端 E2E 降级为人工待办 | 🛑 报告写「前端未自动覆盖」（禁止因本地无 skill 文件提前降级） |
| dev server 起不来 | 换 `app_launch` 命令重试 1 次 | 🛑 前端层降级 + 报告写明启动失败原因 |
| 测试红 = 实现 bug | 改实现重跑（见下方代码改动硬边界） | 🛑 3 轮仍红 → 停下，报告列「未修复项 + 原因」，提示人介入 |
| 变异自检后测试仍绿 | 标记该测试 INVALID | 🛑 报告写「无效测试，建议删除」，不计入有效覆盖 |

**🛑 自动改实现代码的硬边界**（本 skill blast-radius 最大的动作）：

- 只改 **AC 明确要求**的行为；AC 没说的改测试，**不改实现**。
- 每次实现改动**独立 commit**（可单独 review / 回滚），不与测试改动混提。
- **3 轮未修复必停**——不无限重试、不为了变绿弱化 AC。
- 变异自检的改坏代码只在 **worktree 隔离**里进行，用完即丢，主分支不受影响。

## 工作流

### Phase 1：批量探基建 + 起应用

**一次性并行探测所有基建**，不逐项串行确认——把 N 次顺序检查合并成一个 pass，结果汇总后再决定降级策略。

并行探测以下四项：

| 基建项 | 探测方式 |
|--------|---------|
| 测试 DB | 尝试建立测试连接 / 建临时库 |
| API 测试客户端 | 检测 httpx / FastAPI TestClient 可用性 |
| 缓存 | 若项目使用则检测连接 |
| 前端驱动 | 通过 `Skill tool` 调用 `web-access`（见下方规则） |

> 各基建项缺失时的降级处置统一见上方 [§🛑 安全边界与降级处置](#-安全边界与降级处置)，不在此重复。

**汇总规则**：四项全部探测完毕后，一次性输出可用 / 降级清单，再进入步骤 2。不逐项报告、不在探测中途停等用户。

**起应用**（前端 E2E 可用时才需要）：按 `app_launch` 探测/启动 backend + frontend dev server，记录访问 URL；测完负责关闭。

**🛑 缺关键基建 → 汇总上报 + 降级**：按 §🛑 安全边界与降级处置 表逐项降级；一律写进报告（如"前端未自动覆盖"/"后端 DB 测试未覆盖"），**不静默跳过**。

> **web-access 探测规则**（重要）：`web-access` 是**全局 skill**，不在项目 `.claude/skills/` 目录下，**不能用文件系统检测**。正确做法：直接通过 `Skill tool` 调用 `web-access`；若 Skill tool 报错"skill not found"才判定不可用并降级。禁止因本地目录没有 skill 文件就提前降级。

### Phase 2：RBT 规划

1. 读各 task 文档的 **Acceptance Criteria** + plan 的 **Provides/Consumes**，枚举**跨层 E2E 场景**（单 task 写不出的那种，如"前端登录 → 会话落 DB → /me 返回用户"）。
2. 每个场景标注：**触发 → 经过哪些层 → 每层断言什么真实状态**（DB 行 / 缓存条目 / API 信封字段 / 前端 UI）。
3. 按**风险**排优先级（破坏性、数据正确性、安全、核心流程优先）。
4. 产出 `work_dir/_test-plan.md`（人可读：场景 + 层 + 断言 + 优先级）。

### Phase 3：写 + 跑测试（分层）

- **后端层**（pytest + httpx/TestClient）：断言 API 信封 + DB 行 + 缓存，走真实链路、跨层不 mock。
- **前端 E2E 层**（用 `web-access` skill 驱动真实浏览器）：走完整用户流程，例如：打开 `/login` → 输入预置账号 → 提交 → 跳首页 → **断言页面显示用户名 X**（显式断言，不只是截图）。
  - **每条成功流程必须落盘截图**：web-access 截图后，**立即用 `Write` 工具**把图片字节写入 `evidence_dir/{scenario}-ok.png`（`evidence_dir` 需提前 `mkdir -p` 确保存在）。在报告里用相对路径链接该文件。截图仅出现在会话里不算落盘，报告里链接不到文件视为未完成。
- 跑到**全绿**，失败走以下修复循环（最多 3 轮）：

  ```
  失败
   → 判断是测试写错 还是 实现 bug
   → 测试写错：修测试，重跑
   → 实现 bug：直接修改实现代码（遵守 §🛑 自动改实现代码的硬边界：仅改 AC 要求、独立 commit），重跑受影响的测试
   → 修完重跑全绿 → 继续
   → 🛑 3 轮后仍红 → 停下，在报告里列出"未修复项 + 原因"，提示用户介入
  ```

  **判断实现 bug vs 测试错误的标准**：AC 明确要求某行为 → 实现没做到 → 实现 bug，直接改实现；AC 没说 / 测试断言写错了 → 改测试。不要为了让测试通过而把 AC 要求的行为改掉。

### Phase 4：变异自检（parallel subagents，替代人工闸 —— 仅代码级测试）

每个高优先级后端测试**各自派一个 worktree 隔离 subagent 并行执行**——不再串行"改坏→跑→还原"，而是利用 worktree 隔离天然避免代码污染，用完即丢。

**subagent 调度方式**：
- 用 `Agent` 工具，对每个高优先级测试各派一个 subagent，参数：`isolation: "worktree"` + `run_in_background: true`
- 主 agent 不等待，所有 subagent 并行运行

**每个 mutation subagent 的任务**：
```
1. 定位目标测试断言的被测代码行（如：返回值、条件分支、状态写入）
2. 故意改坏该行为（如：返回值取反、删除关键写入、改变状态值）
3. 只跑该条测试：uv run pytest {test_file}::{test_name} -x
4. 返回结果：{ test: "{name}", mutated_line: "file:line", result: "RED|GREEN", verdict: "VALID|INVALID" }
   - RED = 测试检测到变异 → 测试有效（VALID）
   - GREEN = 测试未检测到变异 → 测试无效（INVALID），建议丢弃
```

**注意**：subagent 在 worktree 里修改代码后**不需要还原**——worktree 隔离，用完即丢，主分支代码不受影响。

**主 agent 收取所有 subagent 结果后**：
- VALID 测试：保留，列入报告
- INVALID 测试（变异后仍绿）：标记"无效测试"，丢弃或注明"建议删除"

**前端 E2E 不做变异**（成本高）：信任锚点 = **成功截图 + 显式 UI 断言**。报告须注明"前端层靠截图 + 断言，未做变异自检"——避免给前端假绿的错觉。

### Phase 5：报告

写 `work_dir/_test-report.md`（人读，是你不跑测试也能"心理有底"的入口）：

- 每层覆盖了什么、对应哪些 AC
- RBT 优先级清单（覆盖了哪些、砍了哪些及原因）
- **变异自检结果**（哪些测试证明能抓 bug，哪些被判无效）
- **前端截图链接**（evidence_dir）
- **未覆盖项 + 原因**（如基建缺失降级的部分）

可选：在各 task 文档「变更叙事」补一行"E2E 覆盖：见 `_test-report.md`"。

## ⛔ 反模式红灯（出现任一 = 测试不可信，必须纠正）

- ⛔ **假绿**：测试一直绿就当通过。必须过变异自检证明它真能抓 bug，测不到的判 INVALID 丢弃。
- ⛔ **迁就测试删 AC**：为让测试变绿把 AC 要求的行为删掉/弱化。测出实现 bug 应改实现（见 §🛑 自动改实现代码的硬边界），不是改 AC。
- ⛔ **静默跳过**：基建缺失就悄悄不测。必须降级 + 报告写明未覆盖项（见 §🛑 安全边界与降级处置）。
- ⛔ **前端假绿**：把"截了图"当通过。前端必须有显式 UI 断言；前端不做变异自检，报告须注明信任锚点 = 截图 + 断言。
- ⛔ **截图只在会话**：web-access 截图不落盘。必须 `Write` 到 `evidence_dir`，报告链接不到文件即视为未完成。
- ⛔ **越界测试**：刷覆盖率、重复 vj-work 的 per-task 单元测试、测框架/库/DB 内部（详见铁律的 RBT / 不重复原则）。
